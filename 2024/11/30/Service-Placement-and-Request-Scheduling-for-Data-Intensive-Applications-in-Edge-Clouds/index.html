Service Placement and Request Scheduling for Data-Intensive Applications in Edge Clouds

Problem Formulation

  • 假设有个边缘云，种服务，服务被放置在边缘云或云端。

    用户所提出的任务请求首先会发送给离他最近的边缘云，然后该请求会被路由到能处理该请求的服务所在的边缘云或云端。

    • 在这个过程中，需要消耗的资源包括：
      • 将任务请求的数据由用户设备传输到边缘云所需的传输资源
      • 边缘云的计算资源
      • 将任务请求的数据由边缘云传输到边缘云，这篇论文对其忽略不计了。

  • 这篇论文假设服务是可以随着时间变化放置在不同的边缘云上的，亦即服务是可以进行迁移的。所以这篇论文所采用的服务放置模型是基于 “frame” 和 “slot” 的。

    • 在 “frame” 范围内对服务放置进行确定
    • 在 “slot” 范围内对用户提交的任务请求的调度进行确定

    对提交给边缘云的请求服务的用户请求，记为，在 frame内的平均提交率为，在 slot内的实际到达请求数为。

    对于请求，还有可能存在边缘云是否 “permitted”，用来表示，表示允许，否则表示不允许。

    • 先在 “frame” 范围内确定服务放置，与 “frame” 有关的参数会用来标记。
    • 然后在 “slot” 范围内确定请求调度，与 “slot” 有关的参数会用来标记。

Service Placement Model

Request Scheduling Model

  • 在 slot 内，请求是否被边缘云处理用来表示，表示被处理，否则表示不被处理。
  • 在 slot 内，请求被边缘云实际处理的数量用来表示。
